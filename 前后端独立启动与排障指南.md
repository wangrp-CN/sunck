# 前后端独立启动与排障指南

> 本指南用于在本地（macOS / PyCharm）独立启动涉铁工程智能监控平台的后端（FastAPI）与前端（Vite/Vue），并处理最常见的端口冲突问题。

---

## 一、端口约定

| 服务 | 默认端口 | 配置文件/来源 | 说明 |
|---|---|---|---|
| 后端 | `8000` | `rail_monitor/.env` 的 `PORT=8000` | 可在启动参数中覆盖 |
| 前端 | `5173` | `rail_monitor/web/vite.config.ts` 的 `server.port=5173` | 开发服务器端口 |
| PostgreSQL | `5432` | `rail_monitor/.env` | 数据库，启动时即可用，后端启动不依赖它 |
| Redis | `6379` | `rail_monitor/.env` | 缓存，启动时即可用，后端启动不依赖它 |
| Mosquitto MQTT | `1883` | `rail_monitor/.env` | 后端启动会尝试连接，失败仅告警，不阻塞启动 |

**启动顺序建议**：先启后端，再启前端。前端 `/api` 代理到 `http://127.0.0.1:8000`，后端需要先占住 8000。

## 二、后端独立启动（3 种方式）

### 方式 A：命令行（推荐、最稳）

```bash
cd /Users/wangpeng/PycharmProjects/RimpCode/rail_monitor
.venv/bin/uvicorn app.main:app --host 0.0.0.0 --port 8000 --reload
```

若端口冲突，可临时换端口：

```bash
.venv/bin/uvicorn app.main:app --host 0.0.0.0 --port 8001 --reload
```

> 注意：前端 `vite.config.ts` 默认代理 `/api` 到 8000，若后端端口改了，要同步修改前端代理配置，或改用命令行参数 `npm run dev -- --port 5173` 不变地保持前端端口。

### 方式 B：PyCharm Run 配置

项目已内置 `.idea/runConfigurations/RailMonitor.xml`，配置如下：

- **Type**：`PythonConfigurationType`（不是 `PythonConfiguration`，后者会报配置损坏）
- **Script**：`$PROJECT_DIR$/.venv/bin/uvicorn`
- **Parameters**：`app.main:app --host 0.0.0.0 --port 8000 --reload`
- **Working dir**：`$PROJECT_DIR$`
- **SDK**：`$PROJECT_DIR$/.venv/bin/python`
- **Env file**：`$PROJECT_DIR$/.env`

使用步骤：

1. 在 PyCharm 打开目录：`/Users/wangpeng/PycharmProjects/RimpCode/rail_monitor`（**不是** RimpCode 根目录）。
2. 右上角 Run 下拉框选 **RailMonitor** → 点击绿色运行按钮。
3. 控制台看到 `Uvicorn running on http://0.0.0.0:8000` 即成功。

### 方式 C：模块入口（兜底）

```bash
cd /Users/wangpeng/PycharmProjects/RimpCode/rail_monitor
.venv/bin/python -m rail_monitor
# 或
python app/main.py
```

项目根目录下的 `__main__.py` 会转发给 uvicorn。

## 三、前端独立启动（2 种方式）

### 方式 A：命令行（推荐）

```bash
cd /Users/wangpeng/PycharmProjects/RimpCode/rail_monitor/web
npm run dev
# 或显式指定 host/port
npm run dev -- --host 0.0.0.0 --port 5173
```

> `--` 用于把 `--host`/`--port` 参数透传给 `vite`，而不是给 `npm`。

### 方式 B：PyCharm Run 配置

项目已内置 `.idea/runConfigurations/WebDev.xml`，配置如下：

- **Type**：`npm`
- **Command**：`run`
- **Scripts**：`dev`
- **Working directory**：`$PROJECT_DIR$/web`
- **Node interpreter**：`project`（使用项目 node 版本）
- **Package manager**：`npm`

正常情况：在 PyCharm 打开 `rail_monitor` 目录后，右上角 Run 下拉框应直接出现 **Web Dev**，点击运行即可。

**如果你看到 Web Dev 被归类在 Python 节点下，或报“请指定脚本名称”（如你的截图），说明当前配置被误改成 Python 类型，或者 PyCharm 没有读取项目文件。请按以下方式重建：**

1. 点击右上角运行配置下拉框 → **编辑配置...**（Edit Configurations...）。
2. 在左侧列表中找到当前错误的 **Web Dev** → 点击 **`-` 删除**。
3. 点击左上角 **+ → npm**。
4. 在右侧表单填写：
   - **名称**：`Web Dev`
   - **Command**：`run`
   - **Scripts**：`dev`
   - **Working directory**：点击文件夹图标，选择 `rail_monitor/web`
   - **Node interpreter**：`Project`（即使用项目 node 版本；若下拉框里没有，见下方“Node 解释器路径设置”）
   - **Package manager**：`npm`
   - **Environment**：留空即可
5. 右下角勾选 **“存储为项目文件(S)”**（Store as project file），确保写入 `.idea/runConfigurations/WebDev.xml`。
6. 点击 **确定**。
7. 回到右上角 Run 下拉框，选 **Web Dev** → 点击绿色运行按钮。

正确启动后，PyCharm 运行窗口应显示：

```
> rail-monitor-web@0.1.0 dev
> vite

  VITE v5.4.21  ready in xxx ms

  ➜  Local:   http://localhost:5173/
```

> **重要提示**：前端 Vite 启动成功仅代表前端服务可用。如果控制台出现 `[vite] http proxy error: /api/v1/auth/captcha` 红色报错，说明 **后端未启动**，前端代理不到 8000 端口。此时请先启动后端（见第二节），刷新浏览器页面即可。

### Node 解释器路径设置（关键）

**Node 解释器是什么？** 它就是一个可执行文件，在 macOS 上是 `Mach-O 64-bit executable` 二进制，文件名通常叫 `node`，没有图标，双击不会打开。你可以用终端验证：

```bash
# 查看某个路径的 node 文件类型
file /Users/wangpeng/.workbuddy/binaries/node/versions/22.22.2/bin/node
# 输出示例：Mach-O 64-bit executable arm64

# 查看版本
/Users/wangpeng/.workbuddy/binaries/node/versions/22.22.2/bin/node --version
# 输出：v22.22.2
```

**如何找到本机所有 Node 二进制路径？**

```bash
which -a node          # 按 PATH 顺序列出所有 node
command -v node        # 当前默认 node
ls -la /opt/homebrew/bin/node   # 系统 Homebrew Node
ls -la /Users/wangpeng/.workbuddy/binaries/node/versions/22.22.2/bin/node  # 受管 Node
```

**PyCharm 中指定 Node 二进制路径的步骤：**

1. 在 `Web Dev` 配置面板，点击 **Node 运行时** 右侧的 `...` 按钮。
2. 在弹出窗口中点击左上角 **+ → 添加 Node 解释器**。
3. 选择 **Node 二进制文件路径**（Node.js interpreter path），不是 npm 路径。
4. 在文件选择器中，按 `Cmd + Shift + G` 弹出“前往文件夹”，粘贴：
   ```
   /Users/wangpeng/.workbuddy/binaries/node/versions/22.22.2/bin
   ```
   然后选中 `node` 文件。
5. 点击确定。此时 **Node 运行时** 应显示为项目 node 22.22.2，**软件包管理器** 会跟随该 Node 的 npm（版本 10.9.7）。
6. 点击 **应用 → 确定**。

**为什么推荐用受管 Node 22.22.2 而不是系统 Node 26.4.0？**

- 本项目 `node_modules` 是在 Node 22.22.2 下安装的，Vite 5.4 + Vue 3 工具链在该版本已验证通过。
- Node 26.4.0 太新，部分依赖的二进制包（如 esbuild、fsevents）可能未预编译对应 ABI，运行时容易出现 `NODE_MODULE_VERSION` 不兼容或奇怪警告。
- 系统 npm 11.17.0 官方主要支持 Node 20/22，与 Node 26 搭配不稳健。
- 虽然当前用 Node 26 也能启动前端，但建议统一使用项目受管 Node 22.22.2，避免后续构建/打包异常。

### 方式 C：PyCharm 中前端配置为何会变成 Python 类型？

常见原因：

- 在“编辑配置”时误点了 `+ → Python`，然后给配置起名 `Web Dev`。
- PyCharm 打开的是 `RimpCode` 根目录，而不是 `rail_monitor` 目录，导致项目运行配置文件未被加载。
- 没有勾选“存储为项目文件”，配置只保存在本地 workspace，换环境后失效。

按方式 B 删除旧配置、重新新建 npm 配置即可解决。

## 五、端口冲突：定位与释放

### 5.1 报错样例

```
ERROR:    [Errno 48] error while attempting to bind on address ('0.0.0.0', 8000): address already in use
```

或前端类似：

```
Port 5173 is already in use
```

### 5.2 定位占用进程

**方法 1：按端口查（最准）**

```bash
# 后端端口
lsof -iTCP:8000 -sTCP:LISTEN -n -P

# 前端端口
lsof -iTCP:5173 -sTCP:LISTEN -n -P
```

输出示例：

```
COMMAND     PID   USER   FD   TYPE ...  NAME
python3.1 24014 wangpeng   11u  IPv4 ...  TCP *:8000 (LISTEN)
```

**方法 2：按命令查**

```bash
# 后端相关
pgrep -af "uvicorn app.main:app"

# 前端相关
ps -eo pid,command | grep -E "vite|npm run dev" | grep -v grep
```

### 5.3 安全释放端口

**优先优雅结束**：

```bash
kill -TERM <PID>
```

等待 1~2 秒，再确认：

```bash
lsof -iTCP:8000 -sTCP:LISTEN -n -P
```

若未释放，再强制：

```bash
kill -9 <PID>
```

**一键释放脚本（仅针对本项目，已验证）**：

```bash
# 释放 8000
for pid in $(lsof -t -iTCP:8000 2>/dev/null); do
    kill -TERM "$pid" 2>/dev/null; sleep 1
    ps -p "$pid" >/dev/null 2>&1 && kill -9 "$pid" 2>/dev/null
    echo "释放 8000 PID=$pid"
done

# 释放 5173
for pid in $(lsof -t -iTCP:5173 2>/dev/null); do
    kill -TERM "$pid" 2>/dev/null; sleep 1
    ps -p "$pid" >/dev/null 2>&1 && kill -9 "$pid" 2>/dev/null
    echo "释放 5173 PID=$pid"
done
```

> ⚠️ 不要直接在终端里 `pgrep -af "uvicorn app.main:app" | kill`：若命令字符串本身包含该模式，可能误杀当前 shell。

## 六、常见问题排查

### 1. `[Errno 48] Address already in use`

- 原因：8000 或 5173 已被其他进程（如上次未关闭的 uvicorn/vite、另一个 PyCharm 实例、甚至其他软件）占用。
- 处理：按第 5 节定位并释放。
- 快速规避：临时换端口启动，但不解决根因。

### 2. PyCharm 配置打红叉（如截图中的 `?` 或 `X`）

常见原因：

- PyCharm 打开的是 `RimpCode` 根目录而非 `rail_monitor` 目录。→ **必须打开 `rail_monitor`。**
- Python SDK 未指向 `.venv/bin/python`。→ 在 `Settings → Project → Python Interpreter` 中添加 `.venv`。
- Node.js 未配置或版本不兼容。→ 使用项目已安装的 `node_modules`，或在 `Settings → Languages & Frameworks → Node.js` 中指定 Node 22+。

### 3. 后端启动失败：ImportError / ModuleNotFoundError

- 原因：`.venv` 缺少依赖或依赖版本不对。
- 处理：

```bash
cd /Users/wangpeng/PycharmProjects/RimpCode/rail_monitor
.venv/bin/pip install -r requirements.txt
# 若需严格锁定版本
.venv/bin/pip install -r requirements.lock.txt
```

### 4. 前端启动失败：ERESOLVE / 找不到依赖

- 原因：`node_modules` 缺失或损坏。
- 处理：

```bash
cd /Users/wangpeng/PycharmProjects/RimpCode/rail_monitor/web
rm -rf node_modules package-lock.json
npm install
```

> 项目使用受管 Node 22.22.2；系统 Node 26.4.0 可能不兼容，建议始终使用项目指定的 Node 版本。

### 5. 前端登录后空白或需刷新

- 已修复：确保 `web/src/main.ts` 在 `app.mount` 前 `await router.isReady()`；`auth.store` 中 `loadProfile()` 失败不阻断跳转。
- 若再次遇到：清空浏览器 LocalStorage 中 token 后刷新，或检查后端 `/auth/me` 是否正常。

### 6. 后端日志中 MQTT 连接失败

- 属于**警告**，不影响应用启动。仅实时链路（设备上行/下发）不可用。
- 确认 Mosquitto 已启动：`brew services start mosquitto` 或运行 `scripts/native-services.sh start`。

## 八、速查表

| 现象 | 最可能原因 | 处理 |
